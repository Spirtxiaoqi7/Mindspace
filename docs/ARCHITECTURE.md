# Mindspace 只读拆解与 LangGraph 映射

## 1. 项目实际边界

`Mindspace Launcher` 是 Electron 运维外壳：定位安装根目录、检查模型与端口、启动桌面端、清理已确认身份的服务、显示会话摘要。真正的聊天、RAG、档案回写与 TTS 业务位于相邻的 `Mindspace` 目录。

运行时由三部分组成：

1. Launcher：安装检查、模型下载入口、进程启动与日志入口。
2. Desktop：产品桌面容器。
3. Flask 后端：HTTP API、RAG、生成、会话、档案、知识库和 TTS。

本工程只重构第 3 部分中“一轮聊天”的核心编排，不把 UI、TTS、知识上传和启动器塞进状态图。

## 2. 原聊天链路

原 `/api/chat` 路由按顺序完成：

1. 校验 message、session、round 和 API 配置。
2. 读取并更新会话级角色、协议、分块与调优配置。
3. 用会话配置和当前消息初始化长期档案。
4. 构造 RAG 消融开关。
5. 知识库召回：向量 + BM25 + RRF + 角色名称增益。
6. 对话召回：会话隔离 + 角色名称增益。
7. 对两类结果施加轮次/物理时间指数衰减并排序。
8. 加载最近 10 条消息与结构化关系状态。
9. 拼装基础 prompt、三份长期档案、当前状态、召回上下文和提交 revision。
10. 调用 OpenAI-compatible chat completions。
11. 先流式提取 `response`，再解析 `json_update` 小幅写回计划。
12. 必要时进行一次协议修复调用。
13. 校验提交计划，或回退旧 writeback 兼容逻辑。
14. 处理明确偏好命令与关系状态变更。
15. 写入会话向量原子、消息、记忆、档案、审计日志。
16. 格式化召回详情并返回 HTTP JSON。

## 3. 原有能力的真实状态

| 能力 | 当前实际行为 | 图中处理 |
|---|---|---|
| 双源 RAG | 知识库与会话历史分别召回 | 两个并行节点，汇合后统一时态重排 |
| 流式响应 | SSE 只发 `start` 和最终完整结果 | 不伪装成 token 流；以后可直接消费图事件 |
| RoleValidator | 类已实现，但主 `generate()` 没有调用其 `validate()` | 变成生成后的必经条件节点 |
| 协议修复 | 可二次调用模型修复四标签 | 有界回路，最多一次，防止无限循环 |
| Skill | API 仅写 `queued_placeholder` 日志，无执行器 | 不纳入本版成功能力，也不在图中伪执行 |
| 状态变更 | 当前用户明确输入或待处理删除事件可触发 | 确定性 guard，模型不能绕过 |
| 档案回写 | 每轮最多三个叶子 Patch | 独立 `validate_json_update`，唯一提交点 |
| 重新生成 | 先删旧向量，再生成；档案写回被禁止 | 图内以 `mode=regenerate` 表达，持久化策略明确 |

## 4. 为什么这样拆

旧路由同时承担协议、业务决策和多种存储写入，一旦后半段失败，前面的档案、向量或日志可能已经发生变化。新图遵守三条边界：

- 模型节点只产生候选结果，不能直接写任何存储。
- 所有不可信结构先经过 Pydantic、路径白名单和 revision 校验。
- 会话写入只发生在最终节点；修复结果和重新生成默认不能修改长期档案。

## 5. 状态所有权

| 状态 | 所有者 | 生命周期 |
|---|---|---|
| `ChatRequest` | API/调用方 | 单轮只读 |
| `ProfileBundle` | 档案仓储 | 单轮快照，含 revisions |
| `knowledge_chunks` | 知识召回节点 | 单轮临时 |
| `chat_chunks` | 会话召回节点 | 单轮临时 |
| `ProtocolOutput` | 协议解析节点 | 通过校验后只读 |
| `JsonUpdateValidation` | JSON 写回策略节点 | 单轮决策 |
| `ChatResponse` | 最终节点 | 对外输出 |

## 6. 外部适配边界

图只依赖 `ports.py` 中的接口。接入现有系统时，应分别实现：

- `RetrieverPort`：包装现有 FAISS、SentenceTransformer、BM25/RRF。
- `ProfileRepositoryPort`：包装三份 JSON 档案的原子写、备份与审计。
- `SessionRepositoryPort`：包装 session JSON 和每会话向量索引。
- `LanguageModelPort`：包装 OpenAI-compatible 模型。
- `RolePolicyPort`：组合正则、语义与历史一致性层。
- `AuditPort`：写入独立审计通道并脱敏密钥。

不要让这些适配器反向决定图的路由；路由属于业务编排层。

## 7. 迁移顺序

1. 先以影子模式运行图：使用同一请求，只比较召回、协议与回复，不写旧数据。
2. 对齐双源召回和时态排序结果。
3. 启用新协议/角色校验，但保持持久化只读。
4. 用临时目录验证档案提交、冲突和幂等性。
5. 最后再让 Flask 路由调用图，并逐项替换旧内联逻辑。

这样可以逐阶段比对，不需要一次性改动原业务项目。
